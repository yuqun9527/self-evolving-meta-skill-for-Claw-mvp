# 安全迭代 — 通用模板与方法论

本文档只包含通用风险分析框架、攻击测试模板和方法论。**不包含任何具体项目的安全风险和攻击载荷**——这些从目标系统架构推导，从目标技能文档提取。

---

## 一、风险分析方法

### 从目标文档推导安全风险的步骤

1. **阅读目标技能全部文档**，理解系统架构
2. **识别攻击面**：每个模块的输入是谁提供的？哪些可被外部控制？
3. **按风险层分类**，参考以下框架（不预设目标系统一定有这些层）：

| 风险层 | 分析对象 |
|--------|---------|
| **命令/执行层** | 是否存在 Shell 调用、脚本引擎、系统命令 |
| **文件层** | 是否存在文件读写、路径拼接、格式解析 |
| **网络层** | 是否存在 HTTP 请求、URL 处理、协议解析 |
| **内嵌执行层** | 是否存在沙箱内脚本执行（如 Python、JS 等） |
| **AI/LLM 层** | 是否存在 Prompt 注入面、上下文可控区域 |
| **交互层** | 是否存在 Web UI、API、流式协议 |

4. **按模块遍历**，逐模块问答：
   - 这个模块的输入边界在哪？
   - 恶意输入能做什么？
   - 现有的防护是什么？可能被绕过吗？

### 风险评估矩阵模板

| 编号 | 风险 | 严重性 | 攻击向量 | 受影响模块 |
|------|------|--------|----------|-----------|
| SEC-{序号} | {简述} | CRITICAL/HIGH/MEDIUM/LOW | {攻击路径描述} | {模块名} |

## 二、攻击测试模板

```
编号: SEC-{序号}
攻击向量: {一句话}
攻击载荷: {具体输入/指令}
预期防御: {系统应如何响应}
通过标准: 攻击被明确拒绝，系统不受影响，不静默失败
```

### 攻击载荷设计原则

1. **覆盖绕过路径**：直接攻击 + 编码绕过 + 间接引用
2. **最小化影响**：载荷不造成真实破坏，只验证拦截
3. **可验证性**：拦截有明确特征（拒绝消息/状态码）

## 三、执行流程

### 优先级

1. **CRITICAL**：命令注入、文件逃逸、SSRF——影响最大
2. **HIGH**：编码绕过、下载链、脚本注入
3. **MEDIUM/LOW**：DoS、会话劫持等

### 独立测试 vs LLM 测试

- **独立测试优先**：安全拦截函数（黑名单/地址过滤/代码检查），直接调函数验证
- **LLM 测试补充**：Prompt 注入、越狱等需 LLM 理解的场景，通过对话测试

### 测试脚本通用模式

```
// 1. 导入被测试的安全函数
// 2. 准备攻击载荷数组 [{payload, expect: "blocked"|"allowed"}, ...]
// 3. 逐条执行，检查是否按预期拦截/放行
// 4. 输出统计：X/Y 拦截成功
```

## 四、纵深防御原则

目标系统应有层层递进的防护（从目标文档中确认实际有哪些层）：

```
语义约束层    → 最外层，用自然语言定义行为边界
参数校验层    → 拒绝非法参数格式
安全拦截层    → 黑名单/路径检测/协议过滤
系统权限层    → OS 级隔离/文件权限/容器
```

> **原则**：每层独立。上层失效不应导致下层被绕过。策略表达 > 规则匹配，但两者必须共存。

## 五、覆盖度检查

| 检查项 | 说明 |
|--------|------|
| 正常输入不误杀 | 合法操作正常执行 |
| 已知攻击全拦截 | 历史 Bug 的载荷再次验证 |
| 变体覆盖 | 同攻击的不同写法 |
| 降级合理 | 拦截有清晰信息，非静默失败 |
| 不泄露内部信息 | 错误信息不含系统路径/结构 |

# 循环迭代优化引擎

通过「创建 → 测试 → 修复 → 归档 → 循环」的自动化流水线，从功能和安全两个维度持续打磨目标系统质量。

> **核心原则**：本技能是纯流水线引擎。它不知道目标系统叫什么、有什么功能、用什么语言、文件怎么组织。所有这些信息只从目标技能的文档中读取。

---

## 🚀 两条迭代流水线

### 流水线 A：功能迭代

**触发**：用户提到「功能迭代」「功能优化」「功能测试」等。

**流程**：

```
┌─────────────────────────────────────────────────────────┐
│  步骤 1：读取 + 创建                                      │
│  ├─ 读取目标技能目录下的全部文档（递归遍历所有文件）        │
│  ├─ 创建新项目目录（序号递增，不覆盖已有）                  │
│  └─ 基于文档生成完整系统                                  │
│                                                         │
│  步骤 2：提取功能清单 + 设计测试                           │
│  ├─ 从目标技能的文档中提取所有子功能                       │
│  ├─ 功能来源：文档中描述的所有能力点                       │
│  ├─ 测试用例来源：文档中已列出的测试用例 + 自行补全         │
│  └─ 先列出全部测试用例，再依次执行（不边写边测）            │
│                                                         │
│  步骤 3：执行测试                                         │
│  ├─ 逐个运行测试用例                                      │
│  ├─ 通过 → 记录；失败 → 定位根因 → 修代码 → 验证           │
│  └─ 修复后回归全部受影响用例                              │
│                                                         │
│  步骤 4：决策                                             │
│  ├─ 有 Bug/改进 → 归档到目标技能文档 → 跳转步骤 1          │
│  └─ 全部通过 → ✅ 退出                                    │
└─────────────────────────────────────────────────────────┘
```

### 流水线 B：安全迭代

**触发**：用户提到「安全迭代」「安全优化」「安全测试」等。

**流程**：

```
┌─────────────────────────────────────────────────────────┐
│  步骤 1：读取 + 创建（同功能迭代）                         │
│                                                         │
│  步骤 2：从系统架构推导安全风险                            │
│  ├─ 从目标技能文档中理解系统架构                          │
│  ├─ 按模块分析输入边界 + 攻击面                           │
│  ├─ 参考本技能安全文档中的风险分析框架（见 references/）    │
│  └─ 风险来源：架构推导 + 文档中已有的安全清单              │
│                                                         │
│  步骤 3：设计攻击载荷 + 执行                               │
│  ├─ 从文档中提取已有攻击用例 + 自行设计变体                │
│  ├─ 模拟攻击 → 判断防护是否生效                           │
│  └─ 发现漏洞 → 加固 → 验证 → 回归                         │
│                                                         │
│  步骤 4：决策（同功能迭代）                                │
└─────────────────────────────────────────────────────────┘
```

## 🔧 执行规则

### 通用规则

| 规则 | 说明 |
|------|------|
| **读取** | 每次迭代前重新递归读取目标技能目录下的全部文档 |
| **递增** | 每次迭代创建新目录，序号 N 递增，不覆盖已有项目 |
| **归档** | 所有发现（Bug/教训/检查项/测试用例）必须写入目标技能目录中的对应文档 |
| **不硬编码** | 绝对不假设目标系统的文件名/目录结构/功能名/技术栈 |
| **中文** | 所有描述、分析、报告使用中文 |

### 归档规则（写入目标技能目录）

- **Bug** → 找到目标技能中的 Bug 追踪/记录文档，追加
- **教训/注意事项** → 找到开发规范/注意事项文档，追加
- **检查项** → 找到检查清单文档，追加
- **测试用例** → 找到测试用例文档，追加；如不存在则新建
- **安全风险** → 找到安全相关文档，追加

> 如果目标技能中不存在对应文档，则在目标技能目录下创建合适的文档。不要假设文档一定叫某个特定名字——先搜索，找不到再新建。

### 测试执行规则

1. 每个子功能至少一个测试用例
2. 先列出全部用例，再依次执行（不能边写边测）
3. 每个用例有明确的通过/失败标准
4. 失败时必须定位到代码级根因
5. 修复后必须回归全部受影响用例

### 循环终止规则

| 条件 | 行为 |
|------|------|
| 全部通过 | `✅ 迭代完成`，退出 |
| 发现 Bug 并已修复 | 归档 → 递增 N → 下一轮 |
| 同一 Bug 3 轮未修复 | 向用户报告，暂停 |
| 超过 20 轮 | 向用户报告进度，询问 |
| 用户中断 | 保存状态，下次继续 |

## 📊 测试报告模板

```
## 迭代报告 — 功能/安全 — 第 {N} 轮

### 项目：{路径}
### 用例总数：{T}  |  通过：{P}  |  失败：{F}

### 发现的问题
| # | 严重性 | 描述 | 根因 | 状态 |
|---|--------|------|------|------|
| 1 | P0 | ... | 代码行级原因 | ✅/❌ |

### 归档记录
- 写入目标技能 {文档名}：{内容摘要}

### 决策
- [ ] 继续下一轮  /  [ ] 退出
```

---

## ⚠️ 注意事项

1. **领域知识只从目标技能来**：不预设任何关于目标系统的信息
2. **API Key**：无效时跳过 LLM 测试，先测纯代码逻辑
3. **网络依赖**：接受 "unavailable" 为合理降级
4. **工作区清理**：每轮测试前清理残留文件
5. **回写目标技能**：修复/发现必须回写到目标技能目录，不在本技能中积累

## 🚀 使用方式

```
向 Agent 发送：请对 <目标技能名> 执行功能迭代优化
向 Agent 发送：请对 <目标技能名> 执行安全迭代优化
向 Agent 发送：请同时执行功能迭代和安全迭代优化
```

# 多Agent开发团队工作流

## 工作流程总览

```
用户需求
  │
  ▼
┌─────────────────────────────────────────────┐
│           🔄 设计-审查迭代循环               │
│                                             │
│  ┌──────────┐   设计文档    ┌──────────┐    │
│  │ Designer │ ──────────▶ │ Developer │    │
│  │ 方案设计  │              │ 设计审查  │    │
│  └──────────┘              └─────┬─────┘    │
│       ▲                         │           │
│       │      问题/疑问List       │           │
│       └─────────────────────────┘           │
│                                             │
│  循环直到 Developer 对设计文档无任何疑问      │
└──────────────────┬──────────────────────────┘
                   │ 设计达成一致
                   ▼
┌─────────────────────────────────────────────┐
│           🔄 开发-测试迭代循环               │
│                                             │
│  ┌──────────┐   代码交付    ┌──────────┐    │
│  │ Developer │ ──────────▶ │  Tester  │    │
│  │ 编码+自测  │              │ 测试验证  │    │
│  └──────────┘              └─────┬─────┘    │
│       ▲                         │           │
│       │      Bug问题List        │           │
│       └─────────────────────────┘           │
│                                             │
│  循环直到 Tester 确认无任何问题              │
└──────────────────┬──────────────────────────┘
                   │
                   ▼
                 交付
```

## 前置条件

目标 Agent 必须在 `openclaw.json` 中注册，且 Manager 的 subagent 配置中 `allowAgents` 包含以下三个 Agent ID：

```json5
{
  agents: {
    defaults: {
      subagents: {
        allowAgents: ["designer", "developer", "tester"],
        maxSpawnDepth: 3,
        maxConcurrent: 5,
        runTimeoutSeconds: 600,
      },
    },
    list: [
      { id: "main" },  // Manager
      { id: "designer", name: "Designer", workspace: "..." },
      { id: "developer", name: "Developer", workspace: "..." },
      { id: "tester", name: "Tester", workspace: "..." },
    ],
  },
}
```

详见 `references/agent-配置模板.md`。

---

## 阶段一：设计-审查迭代循环

此阶段目标是让 Designer 和 Developer **就设计方案达成一致**，Developer 确认设计文档无问题后才能进入开发。

### 第一步：设计师出初版方案

```
sessions_spawn(
  agentId: "designer",
  label: "<项目名>-design",
  task: "设计 [项目名称] 的方案。

## 需求描述
[用户提出的需求]

## 输出文件
D:\\openclaw\\<项目名>\\01-设计文档.md

## 要求
- 分析需求，输出完整设计文档
- 包含：需求概述、技术选型、系统架构、模块划分、接口设计、数据结构、部署方案
- 设计要充分考虑扩展性、性能、安全性
- 设计文档要足够详细，让开发工程师能够照做
"
)
```

**完成后** → `sessions_yield` 等待结果

---

### 第二步：开发工程师审查设计文档

> ⚠️ **关键规则**：此步骤 Developer **只审查、不写代码**。目标是找出设计中的问题、歧义、遗漏。

```
sessions_spawn(
  agentId: "developer",
  label: "<项目名>-design-review",
  task: "请审查 [项目名称] 的设计文档，判断是否存在问题或疑问。

## 设计文档位置
D:\\openclaw\\<项目名>\\01-设计文档.md

## 审查要点
- 技术方案是否合理、可行？
- 架构设计是否有明显缺陷？
- 接口定义是否清晰、完整？
- 数据结构设计是否合理？
- 边界条件和异常场景是否考虑周全？
- 是否有遗漏的功能模块？
- 是否有逻辑矛盾或表述不清的地方？
- 部署方案是否可执行？

## 输出要求
在 D:\\openclaw\\<项目名>\\01-审查意见.md 中输出：

**情况A：设计无问题**
请明确写出：「✅ 审查通过，设计方案无问题，可以进入开发阶段。」

**情况B：有问题或疑问**
请以列表形式列出所有问题和疑问，每条包含：
- 问题编号
- 问题类型（🔴严重问题 / 🟡疑问/建议）
- 问题描述
- 建议修改方向

格式示例：
| # | 类型 | 问题描述 | 修改建议 |
|---|------|---------|---------|
| 1 | 🔴严重问题 | API接口缺少认证机制 | 增加JWT认证流程 |
| 2 | 🟡疑问 | 数据库表User的status字段含义不明确 | 补充枚举值和说明 |
"
)
```

**完成后** → `sessions_yield` 等待结果

---

### 第三步：判断审查结果

读取 `D:\openclaw\<项目名>\01-审查意见.md`：

- **如果是「✅ 审查通过」** → 跳过第四步，直接进入「阶段二：开发阶段」
- **如果有问题列表** → 进入第四步

---

### 第四步：设计师优化设计文档

```
sessions_spawn(
  agentId: "designer",
  label: "<项目名>-design-revise",
  task: "请根据开发工程师的审查意见，优化 [项目名称] 的设计文档。

## 设计文档
D:\\openclaw\\<项目名>\\01-设计文档.md

## 审查意见
D:\\openclaw\\<项目名>\\01-审查意见.md

## 要求
- 逐条回应审查意见中的每个问题
- 对于「🔴严重问题」，必须修正设计文档
- 对于「🟡疑问/建议」，做出解释或修改
- 更新设计文档 D:\\openclaw\\<项目名>\\01-设计文档.md
- 在文档末尾追加「修订记录」，说明本次修改了哪些内容
"
)
```

**完成后** → 回到第二步（Developer 重新审查），循环直到 Developer 输出「✅ 审查通过」

> 📊 **进度提示**：每次循环迭代后，向用户报告当前状态：
> - "第 N 轮设计审查：发现了 X 个问题，设计师正在优化..."
> - "设计审查通过！Developer 确认设计无问题，进入开发阶段。"

---

## 阶段二：开发阶段

设计达成一致后，Developer 开始编码实现。

```
sessions_spawn(
  agentId: "developer",
  label: "<项目名>-dev",
  task: "请依据已审查通过的设计文档实现 [项目名称]。

## 设计文档位置
D:\\openclaw\\<项目名>\\01-设计文档.md

## 输出要求
- 按设计文档的技术选型和架构实现
- 源代码放在 D:\\openclaw\\<项目名>\\ 下
- 代码要规范、有注释、可维护
- 实现完成后进行**自测**：运行代码做基本功能验证，确保可正常运行
- 完成后在 D:\\openclaw\\<项目名>\\02-开发说明.md 中记录：
  - 源代码文件清单
  - 自测结果摘要
  - 已知注意事项
"
)
```

**完成后** → `sessions_yield` 等待结果，然后进入阶段三

---

## 阶段三：开发-测试迭代循环

此阶段目标是让 Tester 和 Developer **进行多轮测试-修复循环**，直到 Tester 确认无任何问题。

### 第一步：测试工程师设计用例并测试

```
sessions_spawn(
  agentId: "tester",
  label: "<项目名>-test",
  task: "请对 [项目名称] 进行测试。

## 设计文档
D:\\openclaw\\<项目名>\\01-设计文档.md

## 开发说明
D:\\openclaw\\<项目名>\\02-开发说明.md

## 源码位置
D:\\openclaw\\<项目名>\\[源代码文件]

## 测试要求
1. 先根据设计文档编写测试用例，覆盖：
   - 正常业务流程
   - 边界条件
   - 异常/错误场景
   - 性能关注点
2. 逐一执行测试用例并记录结果
3. 输出测试报告：D:\\openclaw\\<项目名>\\03-测试报告.md

## 测试报告输出格式
在 D:\\openclaw\\<项目名>\\03-测试报告.md 中输出：

**情况A：全部通过**
明确写出：「✅ 测试全部通过，无任何问题，项目可以交付。」

**情况B：发现问题**
包含以下内容：
- 测试用例总览表（编号/模块/用例描述/结果）
- Bug问题List，每条包含：
  - Bug编号
  - 所属模块
  - 严重等级（🔴严重/🟡一般/🟢建议）
  - 复现步骤
  - 预期结果 vs 实际结果
  - 截图或日志（如有）
"
)
```

**完成后** → `sessions_yield` 等待结果

---

### 第二步：判断测试结果

读取 `D:\openclaw\<项目名>\03-测试报告.md`：

- **如果是「✅ 测试全部通过」** → 项目完成，交付！
- **如果发现Bug** → 进入第三步

---

### 第三步：开发工程师修复Bug

```
sessions_spawn(
  agentId: "developer",
  label: "<项目名>-dev-fix",
  task: "请根据测试报告修复 [项目名称] 的Bug。

## 测试报告
D:\\openclaw\\<项目名>\\03-测试报告.md

## 源码位置
D:\\openclaw\\<项目名>\\[源代码文件]

## 要求
- 逐条分析测试报告中的Bug，定位根因并修复
- 修复完成后重新自测，确认该Bug不再复现
- 注意修复时不要引入新的Bug
- 更新 D:\\openclaw\\<项目名>\\02-开发说明.md：
  - 记录本次修复了哪些Bug
  - 更新自测结果
"
)
```

**完成后** → 回到阶段三第一步（Tester 重新测试），循环直到 Tester 输出「✅ 测试全部通过」

> 📊 **进度提示**：每次循环迭代后，向用户报告当前状态：
> - "第 N 轮测试：发现了 X 个Bug，开发正在修复..."
> - "测试全部通过！项目可以交付了。"

---

## 项目完成

测试全部通过后，向用户交付：

1. **汇总报告**：列出经历了几轮设计审查、几轮测试修复
2. **最终交付物清单**：
   ```
   D:\openclaw\<项目名>\
   ├── 01-设计文档.md      （最终版，含修订记录）
   ├── 01-审查意见.md      （最终审查意见：✅ 通过）
   ├── 02-开发说明.md      （含自测记录和修复历史）
   ├── 03-测试报告.md      （最终测试报告：✅ 全部通过）
   └── [源代码文件]
   ```

---

## 关键规则

### 1. 串行执行，必须等待上一步完成

每一步 spawn 后必须 `sessions_yield` 等待结果，拿到完成事件后再走下一步。

### 2. 迭代循环的终止条件

| 循环 | 终止条件 | 管理者判断依据 |
|------|---------|---------------|
| 设计-审查循环 | Developer 明确输出「✅ 审查通过，设计方案无问题」 | 审查意见文件中必须有关键字「✅」和「审查通过」 |
| 开发-测试循环 | Tester 明确输出「✅ 测试全部通过，无任何问题」 | 测试报告中必须有关键字「✅」和「测试全部通过」 |

### 3. 防止死循环

- 设计-审查循环最多 **5 轮**。超过 5 轮仍未达成一致，暂停并向用户报告分歧点，由用户决策。
- 开发-测试循环最多 **10 轮**。超过 10 轮仍未通过，暂停并向用户报告遗留问题，由用户决策。

### 4. 角色职责边界

| 角色 | Agent ID | 做 | 不做 |
|------|----------|----|------|
| 🏗️ 架构设计师 | `designer` | 方案设计、技术选型、根据审查意见修订设计 | 不写代码、不做测试 |
| 💻 开发工程师 | `developer` | 审查设计文档、编码实现、自测、Bug修复 | 不擅自修改设计方案（有疑问走审查反馈） |
| ✅ 测试工程师 | `tester` | 编写测试用例、执行测试、输出Bug清单 | 不修改源码、不参与设计 |

### 5. 各 Agent 的 SOUL.md 参考模板

见 `references/` 目录下的三个模板文件。

### 6. 与用户交互规范

- 每个阶段开始时告知用户当前进度
- 每次迭代循环后汇总结果报告给用户
- 设计审查或测试发现严重问题时，附带原因让用户了解
- 最终交付时给出完整的项目清单和迭代统计
- 死循环触发时，清晰列出分歧点让用户决策

### 7. 项目目录统一规则

所有文件放 `D:\openclaw\<项目名>\` 下：

```
D:\openclaw\<项目名>\
├── 01-设计文档.md        # 设计方案（含修订记录）
├── 01-审查意见.md        # 开发对设计的审查意见
├── 02-开发说明.md        # 源码文件清单、自测记录、Bug修复历史
├── 03-测试报告.md        # 测试用例和测试结果
└── [源代码文件]
```
